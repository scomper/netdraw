## 二、图标系统核心修复

### 5. 正则误杀内部元素属性（最高优先级BUG）
**根因**：`typeIconHTML` 中 `result.replace(/\swidth="[^"]*"/g,'')` 把 `<rect width="20" height="14">` 的属性也删了
**修复**：只匹配 `<svg>` 标签的属性
```javascript
// 旧：全局替换所有 width/height
result.replace(/\swidth="[^"]*"/g,'')

// 新：只替换 <svg> 标签内的
result.replace(/<svg\b([^>]*)>/, function(m, attrs){
  return '<svg' + attrs.replace(/\s+width="[^"]*"/g,'') + '>';
});
```
**影响**：台式机、交换机、服务器、NAS、防火墙、堡垒机等所有含 `<rect>` 的图标

### 6. 线条图标白色不可见
**根因**：`currentColor` 被替换为节点背景色（白色），导致白色描边在白色画布上不可见
**修复**：`fill="none"` 的图标用品牌色描边，浅色品牌色自动降级为 `#333`

### 7. 图标与背景色互斥判断
新增 `isLightColor()` 函数：
- 背景为浅色时，线条图标自动切换为 `#333`
- 彩色图标例外（不受影响）

## 根因分析：为什么这次有这么多坑？

### 1. **正则边界问题** — 最大的坑
`/\swidth="[^"]*"/g` 这个正则设计时只考虑了 `<svg>` 标签，但实际上会匹配 SVG 内部所有元素的 `width` 属性。这是一个典型的"正则写法不够精确"的问题。

**教训**：修改 HTML/SVG 属性时，一定要限定标签范围，不能用全局匹配。

### 2. **currentColor 替换策略不当**
用 `svg.replace(/currentColor/g, color)` 一次性替换所有 `currentColor`，没有区分描边图标和填充图标。描边图标在浅色背景下不可见。

**教训**：颜色替换前要先判断图标类型（stroke vs fill），分别处理。

### 3. **颜色链路断裂**
自定义节点的颜色链路：`COLORS[n.id] → COLORS[n.type] → COLORS.custom`，缺少 `COLORS[n.iconKey]` 这一环。导致选择了带品牌色的图标，但节点颜色仍然用类型默认色。

**教训**：颜色链路要覆盖所有可能的颜色来源，按优先级排列。

### 4. **设计语言不一致**
三个场景（工具面板、画布、选择器）用了不同的颜色策略，没有统一的设计规范。

**教训**：在开始实现前，先定义好设计语言规范，再逐一对照实现。

### 5. **缺少视觉验证**
代码修改后只做了功能验证（属性检查），没有做视觉验证（截图对比）。

**教训**：图标相关的修改，一定要截图对比前后效果。
